# Switchyard Public Language Strategy

## Status

- `partial`

## What This Means

Switchyard is **not** a fully localized product website today.

The current rule is narrower and more honest:

- the default public front door is **English-first**
- bilingual support remains available through helper pages such as glossary,
  FAQ, and selected reference material
- live/browser reality and workstation-specific proof should stay in proof and
  runbook surfaces instead of rewriting the main landing sentence

## Current English-First Surfaces

- `README.md`
- `docs/README.md`
- `docs/media/30-second-overview.md`
- `docs/first-success.md`
- `docs/public-proof-pack.md`
- `docs/public-distribution-ledger.md`
- `docs/index.html`

## Current Bilingual Support Surfaces

- `docs/glossary.md`
- `docs/faq.md`
- selected deep reference pages where bilingual explanations help maintainers
  or local operators

## Why This Matters

The public front door should help a global developer understand three things
fast:

1. what Switchyard is
2. what it is not
3. where to start first

If the landing path switches languages mid-story, the repo stops feeling like a
trustworthy product front door and starts feeling like an internal note wall.

## Claimable Now

- `English-first public front door`
- `bilingual helper pages for glossary/support where useful`

## Not Claimable Yet

- `full product i18n`
- `fully localized docs corpus`
- `fully translated website`

## Next Reasonable Step

If this lane is pushed further later:

- convert more deep public reference pages to English-first
- keep bilingual support only where it clearly reduces confusion
- do not recreate a mixed-language main landing path
