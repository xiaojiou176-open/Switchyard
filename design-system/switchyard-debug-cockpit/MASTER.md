# Switchyard Debug Workbench Override

## [Confirmed] Override Priority

Read `design-system/MASTER.md` first.

If this file conflicts with the root master, the root master wins.

## [Confirmed] Surface Contract

This page is a read-only diagnosis bench.

It is not:

- a hosted control plane
- a KPI dashboard
- a review shell for external stakeholders

## [Confirmed] Surface-to-Donor Map

| Sub-surface | Primary donor | Allowed borrowing | Frozen exclusions |
| --- | --- | --- | --- |
| Page shell | `Linear` | dense-but-calm shell, evidence-first hierarchy, border-led grouping, long-session readability | no `Mintlify` shell, no `Raycast` launcher shell, no glossy cockpit theater |
| Summary row | `Linear` | first-row truth framing for local materials, current browser, next action | no metric-wall hero, no executive-summary dashboard style |
| Evidence stacks and ladders | `Linear` | panel/list/table discipline, mono evidence labels, calm density | no fake charts, no decorative telemetry, no review-room presentation framing |
| Quick actions or overlays | `Raycast` utility chrome only | compact keyboard-first actions, transient overlays, launcher-like assistive controls | must stay subordinate to evidence flow, must not become the primary navigation model |
| Embedded help/reference blocks | `Linear` shell remains primary | `Mintlify` readability only for subordinate reference text or help excerpts | help must not outrank the current verdict or evidence ladder |

## [Confirmed] Forbidden Inheritances

- no enterprise cockpit nouns or panes such as `Teams`, `Usage`, `Seats`, or
  `Billing`
- no chat-style assistant narration, prompt box hero, or conversational home
  framing
- no docs-first white-shell treatment or public-docs rhythm taking over the
  diagnosis bench
- no fake observability charts, success percentages, or invented health scores

## [Current Default] Drift Questions

Before changing this surface, answer all of these with `yes`:

1. Does the first row still explain local materials, current browser, and next
   diagnostic action before deeper evidence?
2. Are evidence stacks still treated as proof surfaces instead of decorative
   widgets or KPI cards?
3. If quick actions appear, are they transient `Raycast`-style utility chrome
   rather than the shell identity?
4. If help content appears, does it remain subordinate to the evidence flow and
   current verdict?
5. Would a builder read this as a read-only diagnosis bench instead of a hosted
   control plane or external review cockpit?

## [Current Default] Visual Constraints

- dark diagnostic shell
- dense but calm information grouping
- border-led hierarchy over glossy elevation
- no fake observability charts
- no enterprise cockpit theater
